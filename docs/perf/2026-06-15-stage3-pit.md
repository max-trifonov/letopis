# Stage 3 point-in-time perf — snapshots (2026-06-15)

Confirms point-in-time reconstruction (S3-01…03) against NFR-1.5 and NFR-3.4
with measured numbers, per task S3-07. Unlike the stage-2 load test (a full
pipeline run under k6, [`2026-06-14-stage2-load.md`](2026-06-14-stage2-load.md)),
this is a targeted read-path benchmark: the property under test is that
reconstruction cost tracks the snapshot interval, not the history length.

The benchmark is the integration test `TestIntegrationPointInTimePerf`
([`test/integration/pit_perf_test.go`](../../test/integration/pit_perf_test.go)),
run against a real MongoDB via testcontainers so the snapshot lookup and tail
scan use their indexes.

## Verdict

| NFR | Target | Measured | Verdict |
|---|---|---|---|
| NFR-3.4 — long history does not degrade reads | replay bounded by interval, not history length | max tail applied = **99** over 507 reads spanning v1…v10000 (interval 100) | ✅ green |
| NFR-1.5 — point-in-time latency | p99 < 200ms at interval 100 | p50 = **3.14ms**, p99 = **6.14ms** | ✅ green |

The NFR-3.4 result is the load-bearing one: it is a structural assertion (every
read applies ≤ `interval` events), not just a wall-clock number, so it catches a
snapshot-builder regression (S3-02) that a fast p99 alone could hide.

## Method

`seedLongHistory` writes a deep single-entity history directly through the
repositories — 10 000 versions of a counter (`amount == version`) bulk-inserted
via `AppendEvents`, with snapshots materialized at every 100th version exactly
as the builder would (S3-02), and the `cur_*` document at the tip. Bypassing the
ingester lets the test stand up 10k versions in milliseconds instead of 10k
round-trips, while leaving the read path looking at a production-shaped
`sn_*`/`ev_*` layout.

The test then reconstructs at 507 target versions — the boundary cases (a
multiple of the interval, the version just after one, the tip and just below it,
deliberately deep targets at v9000/v9999) plus 500 uniformly-random targets from
a fixed seed — and for each asserts:

- **`events_applied ≤ interval`** (NFR-3.4) regardless of the target's depth;
- the **anchor** is the greatest snapshot at or below the target
  (`snapshot_version == ⌊target/interval⌋·interval`);
- **correctness**: `amount == target` and not deleted — identical to a full
  genesis replay, spot-checked against a snapshot-less `Reader` at v1, v4321 and
  v10000 (path equivalence on the deep history).

Latencies are collected per read; p50/p99 are computed over all 507.

## Results

```
point-in-time over 10000 versions, interval 100: 507 reads,
  max tail applied=99, p50=3.135869ms p99=6.14424ms
```

| | |
|---|---|
| History depth | 10 000 versions, single entity |
| Snapshot interval | 100 |
| Reads sampled | 507 (7 boundary + 500 random, fixed seed) |
| Max tail applied | 99 (= interval − 1, the worst case between two snapshots) |
| p50 latency | 3.14ms |
| p99 latency | 6.14ms |
| Latency budget (NFR-1.5) | 200ms |

A read at v9999 (≈10k versions deep) applies the same ≤99-event tail as one at
v150: the snapshot anchors the replay so depth is invisible to the read. That is
NFR-3.4 made concrete — and the ~50× margin under the NFR-1.5 budget is the
direct consequence.

## Index use (explain)

The two reads on the point-in-time hot path are verified to use their indexes by
the storage suite, not assumed:

- **`Nearest`** (snapshot anchor, S3-01) — `TestIntegrationSnapshotProvisionAndIndex`
  explains the `{eid, v ≤ target}` / sort `v:-1` query and asserts an `IXSCAN`
  over `uniq_snap_eid_v`.
- **Collection stats `last_event_at`** (S3-04) —
  `TestIntegrationStatsLastEventNoBlockingSort` explains the reverse-`$natural`
  limit-1 read and asserts the plan has **no blocking `SORT`** stage, so the
  "cheap counters" promise (FR-3.6) holds on a large `ev_*`.

The tail scan itself rides the `{eid:1, v:1}` unique event index
(`uniq_eid_v`), the same index already explained by
`TestIntegrationLastEventUsesIndex`.

## Reproducing

```sh
go test -tags integration -race -run TestIntegrationPointInTimePerf ./test/integration/ -v
```

The p99 threshold is asserted in the test, so a regression past 200ms — or a
builder change that lets a tail exceed the interval — fails CI like any other
test.
