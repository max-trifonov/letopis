# Stage 2 load test — baseline (2026-06-14)

Confirms the async pipeline (S2-01…06) against NFR-1 with measured numbers, per
task S2-07. Scenarios and harness live in [`test/load`](../../test/load);
CPU-side micro-benchmarks in `internal/diff/bench_test.go` and
`internal/service/bench_test.go`.

## Verdict

| NFR | Target | Measured | Verdict |
|---|---|---|---|
| NFR-1.1 — async accept latency (`202`) | p50<2ms, p99<10ms | durable p50=0.37ms / p99=1.78ms; fast p50=0.21ms / p99=5.35ms | ✅ green |
| NFR-1.2 — strict accept latency (`201`, committed) | p99<50ms | p50=5.26ms / p99=15.27ms | ✅ green |
| NFR-1.3 — async throughput, ~2KB event | ≥10 000 events/s | fast: 10 000/s accepted, 0 backpressure; durable: ~5 000/s sustained drain (write-bound) | ⚠️ green for fast; documented deviation for durable (see below) |
| NFR-1.6 — processing lag `202`→stored | p99<1s | 0.02–0.57s at sustained nominal load (4 000/s durable) | ✅ green |
| NFR-2.3 — overload behaviour | `429`+`Retry-After`, no loss/hang | 831 940 × `429`, 100% with `Retry-After`, 0 responses outside {202,429} | ✅ green |

A correctness bug was found and fixed during this run (see
[Bug found](#bug-found-and-fixed-bson-container-types)); the numbers above are
post-fix.

## Environment

Single host, all components co-located via Docker Compose
(`test/load/docker-compose.load.yml`) — a deliberately conservative setup: the
load generator competes with the service and infrastructure for the same cores,
so the throughput figures are a floor, not a ceiling.

| | |
|---|---|
| CPU | AMD Ryzen 7 8845H, 16 threads |
| Service | Letopis `role=all`, 16 queue shards, built from source (Go 1.26) |
| Storage | MongoDB 7, Redis 7 (containers, default tuning, no replica set) |
| Queue | `redis-streams` (durable), in-process memory (fast) |
| Generator | k6 (container, same host) |
| Event | ready ~2KB diff (1924-byte queue payload), 5 000 entities across 16 shards |

A *ready diff* (not full state) is posted so diff-computation CPU stays out of
the accept path (NFR-1.1); each entity is seeded once so the diff applies and
the event reaches Mongo (the precondition for the NFR-1.6 lag).

## Results by scenario

### NFR-1.1 / NFR-1.3 — durable accept

| Arrival rate | p50 | p99 | backpressure | note |
|---|---|---|---|---|
| 4 000/s | 0.37ms | 1.78ms | 0% | sustained; queue depth ≤1.3k, all events land |
| 7 000/s | 0.91ms | 7.65ms | 29% | above worker drain; queue pinned at `max_depth`, excess shed as `429` |

Accept latency is within NFR-1.1 at both rates. The 7 000/s backpressure is the
worker-drain ceiling, not an accept-path problem — see the deviation note.

### NFR-1.1 / NFR-1.3 — fast accept

| Arrival rate | p50 | p99 | backpressure |
|---|---|---|---|
| 10 000/s | 0.21ms | 5.35ms | 0% (queue depth stayed 0) |

Fast sustains the NFR-1.3 target with no backpressure and sub-ms median: the
accept is a channel send and the batcher flushes via `insertMany`. Effective
req/s in the harness (~9k) is capped by the co-located generator (k6 reported
`dropped_iterations` from VU starvation on tail latency), not the service —
queue depth never grew.

### NFR-1.2 — strict

50 VUs, 30s: p50=5.26ms, **p99=15.27ms**, p95=10.62ms, 0 failures, ~7 500
committed writes/s. Comfortably under the 50ms budget.

### NFR-1.6 — processing lag

Sampled `letopis_consumer_lag_seconds{queue="durable"}` during the sustained
4 000/s run: oscillated 0.02–0.57s with queue depth ≤1.3k, draining to 0 within
~3s of load stopping. Under p99<1s at nominal load. (At 7 000/s the lag climbed
to 1–8s because the queue was intentionally driven past capacity — that is the
overload regime NFR-2.3 covers, not the nominal regime NFR-1.6 measures.)

### NFR-2.3 — overload

40 000/s for 30s against a durable queue with `max_depth=20000`: 831 940
responses were `429`, **every one carried `Retry-After`**, and **no response
fell outside {202, 429}** — no 5xx, no hangs, no silent drops. Backpressure sheds
load gracefully and the worker drains the backlog once the surge ends.

## CPU micro-benchmarks

`make bench` (no `-race`; AMD Ryzen 7 8845H):

| Benchmark | ns/op | B/op | allocs/op | What it bounds |
|---|---|---|---|---|
| `diff.Diff` (~2KB doc) | 6 184 | 3 204 | 81 | full-state ingest (derive diff) |
| `diff.Apply` | 1 526 | 2 536 | 21 | worker apply of a ready diff |
| `diff.Canonicalize` | 2 891 | 1 665 | 43 | hash-chain canonical form (later) |
| `service.Encode` (task→payload) | 4 520 | 2 419 | 3 | per-accept serialization (NFR-1.1) |
| `service.Decode` (payload→task) | 19 238 | 6 888 | 144 | per-event worker decode |
| `service.planEvent` | 5 771 | 8 528 | 71 | worker per-event planning (apply + build) |

These confirm the accept path is dominated by I/O, not CPU: encoding a 2KB task
is ~4.5µs, three orders of magnitude under the measured 202 latency.

## Deviation: durable throughput (NFR-1.3)

**Observed.** Durable sustains ~5 000 events/s of *committed* drain on this host
(196 004 events written in 40s while shedding the surplus of a 7 000/s offer);
above that the queue fills to `max_depth` and the accept path correctly returns
`429`. Fast meets the ≥10 000/s target outright.

**Root cause.** The durable worker (`internal/worker/worker.go`) applies one
delivery at a time: `AppendEvent` + a `cur_*` `Upsert`, i.e. two Mongo round
trips per event, parallelised only across the 16 shard goroutines. Fast hits the
target because it batches a flush into a single `insertMany` per collection
(`service.IngestBatch`). So the gap is structural — durable trades throughput for
its at-least-once, ordered, per-event guarantee — not a misconfiguration.

**Assessment.** NFR-1.3 specifies "async" without pinning a mode; the fast path
satisfies it. Durable's ceiling is acceptable for stage 2: backpressure (NFR-2.3)
protects the service under overload, and nothing is lost. Closing the gap (a
durable batch-commit path that preserves per-entity order and at-least-once) is a
worthwhile later optimisation but out of S2-07 scope. The numbers here are also
a floor — Mongo and the generator share the host; a dedicated Mongo and a
separate load box would lift the durable ceiling.

## Bug found and fixed: BSON container types

The load test surfaced a correctness bug, not just numbers. **Durable/fast
`/diff` writes failed for any diff touching a nested object or array.** A
ready diff against a stored entity returned a failed ticket:

```
service: diff does not apply to current state:
diff: type mismatch at path: "items.2.qty" expects an array
```

**Cause.** The Mongo driver decodes embedded documents into `bson.D` and arrays
into `bson.A` when the destination is an `any`. The diff engine works on the
neutral JSON representation (`map[string]any`, `[]any`, …) it documents as its
contract, and its `Apply` type-assertions reject those named BSON types. So the
current state loaded from `cur_*` could not be navigated below the top level;
only top-level scalar diffs applied. The async path was the first consumer to
apply a client diff against a Mongo-loaded base, so earlier stages never hit it.

**Fix.** `internal/storage/mongo/mapping.go` now canonicalises decoded values
back to neutral JSON on read (`canonicalJSON`), applied to current-state `State`
and to event change `Old`/`New` (the latter for the upcoming reconstruction and
JSON-Patch consumers). Storage is the correct seam: its job is the `domain↔BSON`
mapping, and "neutral JSON types" is the documented domain contract. Covered by
`TestCurrentStateDecodesToNeutralJSON` (mapping_test.go), written first to
reproduce the failure. Post-fix, 196 004 durable diffs with nested objects and
arrays applied and landed in one 40s run.

## Reproducing

```sh
make load-up                                   # build + start service, Mongo, Redis
make load-durable                              # NFR-1.1 / NFR-1.3 (durable)
LETOPIS_RATE=4000 make load-durable            # sustained rate for the NFR-1.6 lag
make load-fast load-strict load-overload
make load-down
make bench                                     # CPU micro-benchmarks
```

Scrape the lag mid-run:

```sh
curl -s localhost:8080/metrics | grep -E 'letopis_(consumer_lag_seconds|queue_depth)'
```

Thresholds are encoded in the k6 scripts, so a non-zero k6 exit means a missed
NFR. Note `make load-durable` at the default 10 000/s exits non-zero *by design*:
that rate is above the durable worker drain, so backpressure trips its
`http_req_failed` threshold — the accept-latency thresholds still pass. Use a
sustainable rate (`LETOPIS_RATE=4000`) for a clean durable pass.
