# Концепции

На этой странице описаны ключевые понятия Letopis. Прочитайте её перед изучением API.

---

## Коллекции и сущности

**Коллекция** — именованная группа сущностей одного типа, например `crm.deals`, `docs.contracts` или `payments.invoices`. Имена коллекций — строчные, разделённые точкой идентификаторы (`^[a-z0-9]+(?:\.[a-z0-9]+)*$`). Letopis не интерпретирует смысл имени — это просто пространство имён.

**Сущность** — отдельный отслеживаемый объект внутри коллекции, идентифицируемый строкой `entity_id`, уникальной в рамках коллекции. Letopis хранит `entity_id` как непрозрачную строку.

Коллекции и соответствующие физические коллекции MongoDB создаются автоматически при первой записи, если `collections.auto_create: true` (по умолчанию). Отключите это, чтобы требовать явного `PUT /collections/{c}/config` перед записью.

## События и диффы

Каждое изменение сущности записывается как **событие**. Событие содержит:

- **Дифф** — список изменений на уровне полей (`add`, `change`, `remove`) с путём, старым и новым значением.
- **Версию** — монотонно растущее целое число, per-entity, присваиваемое в порядке записи.
- Метаданные: `author_id`, `source`, `ts_source`, `ts_received`, `ts_stored` и произвольный объект `meta`.

Формат диффа — собственный формат Letopis: каждый элемент содержит `{path, op, old, new}`. Поле `old` (отсутствующее в RFC 6902 JSON Patch) делает каждое изменение самодостаточным. Read API может экспортировать диффы в формате JSON Patch (`?format=json-patch`) для совместимости.

### Ingest по полному состоянию vs по диффу

Изменения можно передавать двумя способами:

- **Полное состояние** (`POST /state`): отправляется весь текущий объект; Letopis сам вычисляет дифф относительно последнего известного состояния.
- **Готовый дифф** (`POST /diff`): отправляется список изменений; Letopis валидирует формат и сохраняет как есть.

Оба варианта создают идентичные события в хранилище. Используйте ingest по полному состоянию, когда система-источник не знает, что именно изменилось; ingest по диффу — когда изменения отслеживаете самостоятельно.

## Режимы надёжности

Каждая запись может выбрать один из трёх режимов надёжности — через дефолт коллекции или переопределение на уровне запроса заголовком `X-Letopis-Mode`.

| Режим | Ответ | Гарантия | Применение |
|---|---|---|---|
| `strict` | `201 Created` | Записано в MongoDB до ответа | Аудируемые записи с синхронным подтверждением |
| `durable` | `202 Accepted` + тикет | Помещено в Redis Streams; воркер пишет в MongoDB | По умолчанию; переживает перезапуск API без потери данных |
| `fast` | `202 Accepted` + тикет | In-memory очередь (деградирует до `durable` при разделении ролей) | Максимальная пропускная способность; данные в очереди теряются при крэше |

По умолчанию для новых коллекций используется **`durable`**. Дефолт коллекции меняется через `PUT /collections/{c}/config`.

**Тикеты:** Асинхронные режимы возвращают `ticket_id`. Запрашивайте `GET /tickets/{id}` для отслеживания статуса: `accepted → processing → stored` (или `failed` с причиной). Тикеты истекают через конфигурируемый TTL (по умолчанию 24 ч).

## Слепки и point-in-time реконструкция

Letopis хранит каждое событие append-only. Для сущности на версии 10 000 восстановление состояния путём перебора всех событий с генезиса было бы слишком медленным. **Слепки** решают эту проблему.

Каждые `N` событий (настраивается per-collection; по умолчанию 100) Letopis материализует текущее состояние сущности как слепок. Point-in-time чтение затем воспроизводит только *хвост* событий от ближайшего слепка, не превышающего запрошенную версию. Результат замеров: p99 менее 7 мс для историй с 10 000 версиями.

Слепки создаются по принципу best-effort: они никогда не блокируют путь записи, а их отсутствие влияет только на скорость чтения, но не на корректность.

## Целостность через hash-chain

**Плагин hash-chain** (`hash_chain`) добавляет криптографическую защиту от подделки истории сущности. При включении для коллекции:

```
hash₁ = sha256("letopis:genesis:v1:" + collection_name)  // начальное значение
hash₂ = sha256(hash₁ ‖ canonical(event₁))
hash₃ = sha256(hash₂ ‖ canonical(event₂))
…
```

Каждое событие хранит свой `hash` и `prev_hash`. Поскольку цепочка линейна per-entity, незаметное удаление, вставка или изменение любого события ломает все последующие хэши. Ручка `:verify` обнаруживает первое расхождение.

В каноническую проекцию входят `{op, entity_id, changes, author_id, source, ts_source, flow}`, но не `version` — это позволяет ретраям с другой версией не ломать цепочку.

Hash-цепочки — per-entity. GDPR-удаление (`purge`) уничтожает всю цепочку сущности и фиксирует tombstone; другие сущности не затрагиваются.

## Мультитенантность

Каждый тенант владеет изолированной базой MongoDB (`hm_t_{id}` на кластере по умолчанию или на любом URI/базе, которые вы настроили). Коллекции, события, слепки, правила, тикеты и записи DLQ никогда не смешиваются между тенантами. Тенант определяется **API-ключом**; в URL он не фигурирует.

Внутри базы тенанта каждая логическая коллекция отображается на три физические MongoDB-коллекции:

| Физическая коллекция | Содержимое |
|---|---|
| `ev_{name}` | Append-only события |
| `sn_{name}` | Слепки |
| `cur_{name}` | Текущее материализованное состояние (один документ на сущность) |

Плюс системные коллекции:

| Физическая коллекция | Содержимое |
|---|---|
| `ev__system` | Аудит-лог (изменения конфига, правил, tombstone после purge) |
| `ev__flow` | Активности бизнес-флоу |
| `_rules` | Определения правил |
| `_collections` | Явный конфиг коллекций |
| `_dlq` | Записи dead-letter queue вебхуков |

## Бизнес-флоу и активности

**Активность** — это бизнес-процессное событие, не являющееся изменением сущности, — например, «запущено согласование счёта», «вызван пересчёт», «отправлено уведомление». Активности хранятся в `ev__flow`.

**Флоу** — причинно-связный граф событий и активностей, объединённых общим `flow_id`. Любое событие изменения или активность могут нести блок `flow` с `flow_id`, `caused_by` (ссылки на вышестоящие события или активности) и `step` (произвольная метка). Ссылки сохраняются как есть; Letopis не проверяет, существуют ли referenced-сущности — причинные рёбра через асинхронные системы могут быть «висящими» и разрешаются при чтении.

`GET /flows/{flow_id}` возвращает все узлы в порядке поступления с рёбрами `caused_by`, образующими обходимый DAG.

## Правила и вебхуки

**Правило** привязывается к коллекции и состоит из:

- **Условия** — вложенного дерева комбинаторов `all`/`any`/`not` и листовых операторов (`eq`, `ne`, `in`, `gt`, `gte`, `lt`, `lte`, `regex`, `exists`), вычисляемых по полям события, плюс `changes`-матч с поддержкой glob-путей (`items.*.price`).
- Одного или нескольких **действий**: `webhook` (подписанный HMAC-SHA256 HTTP POST) или `log` (структурированная запись в лог).

Правила вычисляются **после** успешной записи события и никогда не замедляют путь записи. Доставка вебхуков — at-least-once с настраиваемым экспоненциальным backoff. Исчерпанные доставки попадают в DLQ правила, доступный через `GET /collections/{c}/rules/{id}/dlq` и повторно отправляемый через `:redeliver`.

Запрос вебхука подписывается: заголовок `X-HM-Signature` содержит `sha256=` + `hex(HMAC-SHA256(secret, body))`. Секрет разрешается по `secret_ref` из конфига (`webhooks.secrets`), а не хранится в самих правилах.

## Система плагинов

Система плагинов предоставляет три хука в пайплайне ingest:

| Хук | Когда | Что может делать |
|---|---|---|
| `pre-store` | До записи в MongoDB | Обогащать событие, добавлять метаданные, отклонять запись |
| `post-store` | После записи в MongoDB | Реагировать, индексировать, триггерить внешние системы |
| `action` | Когда правило срабатывает | Реализовывать пользовательские типы действий |

Плагины — in-process Go-код, компилируется вместе с сервисом. Регистрируйте их через `pkg/ext.Registry` до запуска сервера. Единственный публичный API — `pkg/ext`; он стабилен по semver. Внутренние пакеты не являются стабильными.

Плагин `hash_chain` поставляется в комплекте и регистрируется по умолчанию (активен только при явном включении per-collection через конфиг).

Ошибка плагина обрабатывается согласно настроенному `fail_mode`:

- `open` (по умолчанию): залогировать ошибку и продолжить — событие записывается без вклада плагина.
- `closed`: отклонить запись с `422 Unprocessable Entity`.
